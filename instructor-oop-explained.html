01 — Overview

Layered Architecture

Your instructor module is a classic Spring Boot REST API following a 5-layer architecture. Each layer has one responsibility — and each maps to a specific Java class in your project.

CLIENT
HTTP Client (Browser / Postman / Frontend) Sends GET, POST, PUT, DELETE requests to /api/instructors
CTRL
InstructorController.java — @RestController Receives HTTP requests, delegates to service, returns HTTP responses
SVC
InstructorService.java — @Service Holds all business logic: ID generation, null-safe updates, DTO ↔ Entity conversion
REPO
InstructorRepository.java — @Repository (JpaRepository) Speaks to the database; Spring auto-generates all SQL for you
DATA
Instructor.java (Entity) + InstructorDTO.java Entity = database table row. DTO = the shape of data sent over the network
DB
MySQL / PostgreSQL — instructor table Persistent storage managed by JPA / Hibernate
↑ Data flows up from DB to Client  |  Requests flow down from Client to DB ↑

03 — OOP: Inheritance

Instructor extends Person

The most visible OOP concept in your code. Instructor inherits from a Person base class, which (based on the import) likely lives in com.drivingschool.common.entity.

🧑 Person (base class)
id · fullName · email · phone
username · password · role
🏫 Instructor (child class)
nic · licenseNumber · specialization
yearsOfExperience · status · joinedDate 
// Child class — Instructor.java
@Data
@EqualsAndHashCode(callSuper = true)
@NoArgsConstructor
@AllArgsConstructor
@Entity
public class Instructor extends Person {

  // Only EXTRA fields go here!
  private String nic;
  private String licenseNumber;

  @Enumerated(EnumType.STRING)
  private Specialization specialization;

  private int yearsOfExperience;
  private String status;
  private String joinedDate;
}

What Instructor INHERITS from Person

Even though they don't appear in Instructor.java, these fields exist on every Instructor object at runtime:

🔑 id (primary key) 👤 fullName 📧 email 📱 phone 🔐 username + password 🎭 role

This is DRY (Don't Repeat Yourself). Other entities like Student or Admin can also extend Person and get the same fields.

⚠️ Why @EqualsAndHashCode(callSuper = true)?

Lombok's @Data generates equals() and hashCode() based only on fields declared in the current class. callSuper = true tells Lombok: also include the parent class fields (like id) when comparing two Instructors. Without it, two instructors with the same nic but different IDs would be considered equal!

04 — OOP: Encapsulation

Data Hidden Behind Getters & Setters

All fields are private. The outside world can only interact through controlled getter/setter methods. Lombok's @Data generates these automatically.

// What you WRITE (InstructorDTO.java)

}

public String getFullName() {
    return this.fullName;
}
public void setFullName(String fullName) {
    this.fullName = fullName;
}
// ... a getter/setter pair for EVERY field

Why Encapsulation Matters

In updateInstructor() in the service, you can see encapsulation protecting data integrity:

// Null-safe update — ONLY changes
// what the caller provided
if (dto.getFullName() != null)
    existing.setFullName(dto.getFullName());
if (dto.getEmail() != null)
    existing.setEmail(dto.getEmail());

Because fields are private, the service is the gatekeeper — it decides what can be changed and under what rules.

05 — OOP: Abstraction

Hiding Complexity Behind Interfaces

Abstraction means: show only what's necessary, hide the implementation. Two places in your code use this powerfully.

1. JpaRepository Interface

You declare the interface with what you want to do. Spring Data JPA writes the actual SQL and Java code behind the scenes.

@Repository
public interface InstructorRepository
    extends JpaRepository<Instructor, String> {

  // Spring generates: SELECT * FROM instructor
  //                  WHERE email = ?
  Instructor findByEmail(String email);

  // And you get these for FREE:
  // findAll(), findById(), save(), deleteById()
}

2. Service as Abstraction Layer

The controller never touches the database. It only calls the service's clean, named methods. All complexity lives inside the service.

// Controller sees this clean API:
instructorService.getAllInstructors();
instructorService.getInstructorById(id);
instructorService.createInstructor(dto);
instructorService.updateInstructor(id, dto);
instructorService.deleteInstructor(id);

// Controller doesn't know about:
// — ID generation logic (I0001, I0002...)
// — password encoding
// — DTO ↔ Entity conversion
06 — OOP: Polymorphism

Same Name, Different Behaviour

Polymorphism = "many forms". It shows up in two ways in your code.

Runtime Polymorphism — Spring DI

When Spring sees @Autowired, it looks up the correct implementation at runtime and injects it. The controller doesn't need to know which InstructorService implementation it's using.

@RestController
public class InstructorController {

  @Autowired  // Spring injects the right impl
  private InstructorService instructorService;
  // Could swap in a mock in tests!
}

Method Overloading — Enum

The Specialization enum has a getLabel() method that returns a different string for each constant — same method, different output. Classic polymorphic behaviour.

Specialization.MOTORCYCLE.getLabel()
// → "Motorcycle"

Specialization.HEAVY_VEHICLE.getLabel()
// → "Heavy Vehicle"

// Same method — 4 different results

Conversion Methods (Overloading Pattern)

The service has two private helpers — convertToDTO(Instructor) and convertToEntity(InstructorDTO). They're complementary transformations: same concept, opposite direction. Used inside map(this::convertToDTO), this is a method reference used polymorphically in a stream.

instructorRepository.findAll()
    .stream()
    .map(this::convertToDTO)  // method reference = polymorphic!
    .collect(Collectors.toList());

08 — Special Types

Enum: Specialization

An enum is a special class where all instances are predefined constants. It's perfect for a fixed set of options like vehicle specializations.

@Getter  // Lombok generates getLabel()
public enum Specialization {

  MOTORCYCLE("Motorcycle"),
  LIGHT_VEHICLE("Light Vehicle"),
  HEAVY_VEHICLE("Heavy Vehicle"),
  BUS("Bus");

  private final String label;

  // Constructor — sets the label
  Specialization(String label) {
    this.label = label;
  }
}

Enum Constants

🏍 MOTORCYCLE 🚗 LIGHT_VEHICLE 🚛 HEAVY_VEHICLE 🚌 BUS

Stored in the database as a STRING (e.g. "MOTORCYCLE") because of @Enumerated(EnumType.STRING). Human-readable and safe even if you reorder the constants later.

Why Use Enum Instead of String?

If specialization was a plain String, someone could set it to "Bicycle" or leave a typo like "motrCycle". An enum prevents invalid values at compile time, not just at runtime.

@PostMapping
public InstructorDTO createInstructor(
    @RequestBody InstructorDTO dto) {

  return instructorService.createInstructor(dto);

// 2. Service handles business logic
public InstructorDTO createInstructor(InstructorDTO dto) {
  // Auto-generate ID: I0001 → I0002 → ...
  Instructor lastIns = repo.findLastManualId();
  String nextId = "I0001";
  if (lastIns != null) {
    int last = Integer.parseInt(lastIns.getId().substring(1));
    nextId = String.format("I%04d", last + 1);
  }
  Instructor entity = convertToEntity(dto);
  entity.setId(nextId);
  entity.setStatus("AVAILABLE");            // default
  entity.setJoinedDate(LocalDate.now().toString()); // today
  return convertToDTO(repo.save(entity)); // 3. → DB
}

Smart ID Generation

// Custom query in InstructorRepository:
@Query(value = "SELECT * FROM instructor WHERE id LIKE 'I%'
         ORDER BY CAST(SUBSTRING(id, 2) AS UNSIGNED) DESC
         LIMIT 1", nativeQuery = true)
Instructor findLastManualId();

// This finds the highest existing ID:
// I0001, I0003, I0007 → returns I0007
// Service then generates I0008
10 — Design Pattern

DTO Pattern — Data Transfer Object

A DTO is a separate object used to transfer data over the network. It's different from your database Entity, and this separation is intentional and important.

Entity vs DTO — What's Different?

FeatureEntityDTO
PurposeMaps to DB tableNetwork transfer
Annotation@Entity@Data
PasswordHashed stringPlain text input
RoleStored in DBSet by client
Sensitive?Yes — never exposeControlled shape

Why Not Just Return the Entity Directly?

If you returned Instructor (the entity) from the API, you would expose:

❌ Hashed password in JSON response ❌ Internal Hibernate proxy metadata ❌ Lazy-loading fields that could break serialization ❌ DB schema structure leaked to client

The DTO is a clean, safe, controlled view of the data. The service's convertToDTO() decides exactly what the client sees.

// convertToDTO() — the "security filter"
private InstructorDTO convertToDTO(Instructor instructor) {
    InstructorDTO dto = new InstructorDTO();
    dto.setId(instructor.getId());
    dto.setFullName(instructor.getFullName());
    dto.setEmail(instructor.getEmail());
    // ⚠️ Notice: password is NOT copied to DTO!
    // The hashed password stays on the server side
    dto.setUsername(instructor.getUsername());
    dto.setRole(instructor.getRole());
    return dto;
}
11 — Error Handling

Custom Exception Class

InstructorException is a custom RuntimeException — this is OOP at its simplest: extending a built-in Java class to give domain-specific meaning to errors.

public class InstructorException
    extends RuntimeException {

  public InstructorException(String message) {
    super(message); // passes to RuntimeException
  }
}

OOP Concepts Used

This tiny class demonstrates Inheritance (extends RuntimeException), Constructor chaining (super(message)), and could be used for Polymorphism — any catch block that catches RuntimeException will also catch InstructorException.
